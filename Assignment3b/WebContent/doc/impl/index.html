DP2-RNS REST API Implementation Documentation


RNS System Design:


The service library has 3 layers: resource, service and storage.

The resource layer has all the HTTP operations required for interacting with the places/connections/vehicles resources. When a request with a body arrives, the body is first checked using the XML validator or JSON interceptor(which increases efficiency, because service layer is not involved).

The jaxb objects from the resource operations(GET/POST/PUT) are converted and added by WADL to the automatically generated schema, which is used by clients.

NOTE: because of the limitations of WADL, elements are not added to the automatically generated schema if the return type from the resource operations if of type "Resource". This means that I cannot set the page header for the POST response (In any case, the user can get the url from the self element of the resource, in the response body)

The service layer is a singleton, and the resource layer uses it by calling the corresponding methods for the resource.

The storage layer is where all the information about places, vehicles is stored. It uses in memory representation of data structures like maps, sets, lists.

The clients interact with the service by using the HTTP operations on the resource layer. Then, based on the operation, the request is forwarded to the service layer, which contacts the Neo4j database or the local storage.

Other implementation information:

  • Runtime exceptions (e.g. from Response objects) are caught
  • Memory leaking is managed by using local variables in methods and by calling close() on the appropriate resources
  • the doc folder and xsd schema are uploaded to WebContent and updated automatically by the ant script, using copyToDir task
  • When a vehicle client calls untrack method on the service, the method PUT was used instead of DELETE because of the presence of a request body (RFC 7231 states that DELETE request body can be ignored by some servers)
  • When a vehicle moves from a place to another one: the path computed for the vehicle from new place to destination can be empty (vehicle client will receive an empty list)
